# v3 PoC — 결정론적 엔티티 그래프 + 그래프 확장 검색

실제 코퍼스(135개 문서)로 Graph RAG의 타당성을 **GPU·외부 API 없이** 검증하는 프로토타입입니다.

관련 문서: [`../docs/v3-planning/01_현행진단리포트.md`](../docs/v3-planning/01_현행진단리포트.md) · [`02_v3_아키텍처_기획서.md`](../docs/v3-planning/02_v3_아키텍처_기획서.md)

## 실행

```bash
cd v3-poc

# 1) 그래프 + 인덱스 구축
python3 poc_graph_rag.py build --corpus ../hf_space_bundle/organized_maintenance_docs_simple

# 2) 통계
python3 poc_graph_rag.py stats

# 3) 검색 — 렉시컬 단독 vs 그래프 확장 나란히 비교
python3 poc_graph_rag.py search "대한항공 홈페이지 관리자 페이지"

# 4) 평가셋 실행
python3 poc_graph_rag.py eval --cases cases.sample.json --out result.json
```

의존성 없음(표준 라이브러리만). `kiwipiepy`가 설치되어 있으면 자동으로 사용하고, 없으면 정규식+n-gram으로 폴백합니다. **위 실측 결과는 폴백 상태에서 나온 것이므로, Kiwi 설치 후 재측정하면 수치가 달라집니다.**

DB는 기본적으로 임시 디렉터리에 생성됩니다(`POC_DB` 환경변수 또는 `--db`로 변경 가능). 네트워크 마운트에서 SQLite 락이 실패하는 것을 피하고, 저장소에 바이너리를 남기지 않기 위함입니다.

## 검증 결과

### H1. 규칙만으로 엔티티를 충분히 추출할 수 있는가 — **⚠️ 부분 확인. 초기 89.8%는 과대평가였음**

```
엔티티가 1개 이상 연결된 청크: 712/793 (89.8%)
```

> **⚠️ 이 수치는 정정되었습니다.** `/v3` 운영 구현(`backend/graph_v3.py`)으로 옮기면서 확인한 결과, 89.8%의 상당 부분은 `기능개선` 이슈유형 하나가 만들어낸 것이었습니다. PoC의 `ISSUE_TERMS["기능개선"]`에 `개선·수정·요청·추가·변경` 같은 범용어가 들어 있어 **793개 중 639개(81%) 청크에 매칭**됐기 때문입니다.
>
> 이런 엔티티는 커버리지 숫자만 올릴 뿐 문서를 전혀 변별하지 못합니다. `기능개선` 태그가 붙었다는 것은 "이 문서는 유지보수 문서다"라는 말과 같습니다.
>
> `graph_v3.py`에서 해당 용어를 `개선·리뉴얼·기능추가`로 좁힌 결과 **실제 커버리지는 42.5%**입니다. 이것이 정직한 수치입니다.
>
> **수정된 결론:** 규칙 기반 추출은 여전히 유효합니다(재현 가능, 재빌드 비용 0, 누락 지점 특정 가능). 다만 **현재 사전으로는 청크의 절반 이하만 의미 있는 엔티티에 연결됩니다.** 사전 확충이 Phase 0의 실질 과제이며, 커버리지를 KPI로 삼을 때는 허브 엔티티를 제외하고 세야 합니다.

비교 기준: 통제 실험(arXiv 2502.11371)에서 **LLM 추출 지식그래프는 정답 엔티티의 약 65.8%만 포착**했습니다.

규칙 기반 추출의 장점은 커버리지 절대값이 아니라 **성질**에 있습니다:

- 100% 재현 가능 (같은 입력 → 같은 그래프)
- 재인덱싱 비용 사실상 0 (135문서 전체 빌드 수 초)
- 누락분이 어디인지 특정 가능 → **사전 확충으로 개선 가능**. LLM 추출은 왜 놓쳤는지 알 수 없음

**Customer 61개, Document 135개는 폴더/파일 구조에서 나온 100% 정확한 엔티티입니다.** LLM이 추출할 이유가 없는 정보입니다.

> 사전은 현재 `SYSTEM_TERMS`(20개) / `ISSUE_TERMS`(6개)에 하드코딩되어 있습니다. 프로덕션에서는 `graph_node_aliases` 테이블 + 관리자 UI로 관리하고, 초기 시드는 Soynlp 코퍼스 마이닝 + 유지보수팀 검수로 확보합니다.

### H2. 그래프 확장이 recall을 올리는가 — **❌ 미입증. 확인된 것은 "재순위화 개선"까지**

**결론을 먼저 명확히 합니다.** 이 PoC에서 **recall 상승을 보여주는 증거는 나오지 않았습니다.**

- `코웨이 SSL 인증서 갱신` 질의에서 도구가 보고한 값은 **"그래프가 새로 끌어온 청크: 0/10"**
- 5건 스모크 테스트에서 **그래프 전용 정답 0건**, Recall@5는 두 모드 모두 1.0으로 포화
- 실제로 개선된 지표는 **MRR (0.52 → 0.77)** — 즉 순위가 올라간 것이지, 없던 정답이 생긴 것이 아닙니다

확인된 것은 **"정답이 후보에는 있었는데 아래에 묻혀 있던 것을 위로 올린다"**까지입니다. 진단 리포트가 지적한 recall 천장(top-3 == top-5) 문제를 그래프가 해결한다는 증거는 **아직 없습니다.**

**이것은 실패가 아니라 예상된 결과입니다.** 재순위화 개선만으로도 체감 품질은 크게 달라지며, recall 개선은 원래 임베딩(KURE-v1)의 몫입니다. 다만 **"Graph RAG를 도입하면 recall이 오른다"는 기대는 현 시점에서 근거가 없으므로 기획에 반영하면 안 됩니다.**

아래는 재순위화 개선의 실측 사례입니다.

**질의: `대한항공 홈페이지 관리자 페이지`**

| 순위 | A. 렉시컬 단독 | B. 그래프 확장 + 고객사 스코핑 |
|---|---|---|
| 1 | `_공통 / SIMPLIFY_CHANGELOG` ❌ | ★ `대한항공 / 디비_접속_정보` |
| 2 | `_공통 / SIMPLIFY_CHANGELOG` ❌ | `대한항공 / HK공통매뉴얼_추가정보` |
| 3 | `대한항공 / HK공통매뉴얼_추가정보` | `대한항공 / 유지보수정보` |
| 4 | `대한항공 / 유지보수정보` | `대한항공 / 유지보수정보` |
| 5 | `대한항공 / 유지보수정보` | `대한항공 / 유지보수정보` |
| 6 | `대한항공 / 유지보수정보` | ★ `대한항공 / HK공통매뉴얼_추가정보` |

★ = 렉시컬 상위 6에 없던 청크

렉시컬 단독은 **1·2위를 저장소 메타 문서(`SIMPLIFY_CHANGELOG`)에 낭비**했습니다. 그래프+스코핑은 상위 6개를 전부 해당 고객사 문서로 채웠고, `대한항공_디비_접속_정보`를 1위로 끌어올렸습니다.

> 정확히 기록하면: `대한항공_디비_접속_정보`는 렉시컬에서도 **10위(점수 19.34)로 존재**했습니다. "렉시컬이 전혀 못 찾았다"고 말하면 과장입니다. 그래프가 한 일은 **발굴이 아니라 재순위화**입니다.

**질의: `코웨이 SSL 인증서 갱신`**

렉시컬 단독은 상위 3개가 `성의교정_카톨릭대학교`, `시도지사협의회` — **질의에 명시된 고객사가 아닌 문서**였습니다. 코웨이 문서는 4위부터 등장. 그래프+스코핑 적용 후 1~5위가 전부 코웨이 문서로 교정되었습니다.

**단, 정직하게 기록해야 할 부작용:** `비슷한 도메인 만료 사례가 있나` 질의에서는 그래프가 상위 6개를 전부 새로 채웠지만, 내용은 `SIMPLIFY_CHANGELOG`(저장소 메타 문서)였습니다. **그래프가 노이즈를 증폭시킨 사례입니다.** 원인은 H3 + 코퍼스 오염(아래 §부수 발견).

### H3. 엣지 제한 없이 확장하면 노이즈가 폭증하는가 — **✅ 확인됨. 그리고 제한만으로는 부족**

**초기 구현(엣지 화이트리스트만 적용)에서 그래프 확장이 상위 8개에 새 청크를 0개 추가했습니다.** 원인 분석:

`SSL인증서` 같은 엔티티 노드가 수백 개 청크에 연결되어 있어, 이를 경유한 2홉 확장이 **코퍼스의 상당 부분을 균일한 점수로 끌어옵니다.** 결과적으로 그래프 런의 순위가 무의미해지고 RRF에서 렉시컬을 희석시키기만 합니다.

**해결 2가지를 추가했고 둘 다 효과가 있었습니다:**

**(1) 허브 페널티** — 엔티티 차수에 로그 역비례하는 감쇠. IDF와 동일한 발상.

```sql
exp_weight = weight / (1 + 0.9 * ln(dst_degree))
```

**(2) 고객사 스코핑** — 질의에 고객사가 링킹되면 해당 고객사 청크를 **별도 런**으로 분리하고 RRF에서 가장 높은 가중치(1.2)를 부여. 유지보수 도메인에서는 "어느 고객사인가"가 거의 항상 1차 필터이므로, 이 신호를 일반 그래프 확장 점수에 묻히게 두면 안 됩니다.

> **설계 교훈:** 그래프 확장은 "켜면 좋아지는 기능"이 아닙니다. **엣지 화이트리스트 + 허브 페널티 + 도메인 특화 스코핑**이 함께 있어야 작동하며, 이 셋이 없으면 순수한 손해입니다. 기획서 §4.2에 반영되었습니다.

### 평가 하네스 스모크 테스트

`cases.sample.json`(5건)으로 `eval` 명령을 검증했습니다:

```
모드            Recall@5   Recall@10     MRR
lexical         1.0000      1.0000  0.5167
graph           1.0000      1.0000  0.7667
```

**이 수치를 성과로 인용하면 안 됩니다.** 5건짜리 손수 만든 케이스이고 Recall@5가 이미 포화(1.0)라 변별력이 없습니다. 의미 있는 것은 **MRR 0.52 → 0.77**(정답이 더 위로 올라옴)이라는 방향성뿐이며, 이마저도 n=5입니다.

하네스가 정상 동작한다는 것과, 상보성 지표(그래프 전용 / 렉시컬 전용 정답 건수)가 집계된다는 것만 확인된 상태입니다. **실제 판단은 150건 평가셋 구축 후에 가능합니다.**

## 부수 발견 — 코퍼스 오염 (즉시 조치 권고)

`hf_space_bundle/organized_maintenance_docs_simple/` 최상위에 **유지보수 콘텐츠가 아닌 저장소 메타 문서**가 섞여 있습니다:

```
SIMPLIFY_CHANGELOG.md
SIMPLIFY_VALIDATION_REPORT.md
READABILITY_CLEANUP_REPORT.md
READABILITY_FINAL_VALIDATION.md
README.md
HK_CUSTOMER_INFO_INDEX.md      ← 고객사 색인. 노이즈가 아닐 수 있어 판단 필요
```

이 문서들이 **여러 질의에서 렉시컬 상위 1~2위를 차지**했습니다. 문서 처리 이력을 기록한 파일이라 도메인 어휘가 대량으로 포함되어 있어 BM25 점수가 높게 나옵니다.

**135개 문서 중 최소 5개가 명백한 저장소 메타 문서이며, 이것만 제외해도 top-3 정확도가 개선될 가능성이 있습니다.** 진단 리포트가 지적한 "실패 케이스 근본원인 분류"(P0)에서 최우선 확인 대상입니다. 알고리즘 개선보다 훨씬 싼 조치입니다.

## Postgres 이식 노트

PoC의 SQLite 재귀 CTE는 **Postgres에서 거의 그대로 동작**합니다.

| PoC (SQLite) | 프로덕션 (Postgres) |
|---|---|
| `postings` 테이블 + 파이썬 BM25 | `tsvector` + GIN 인덱스, `ts_rank_cd` |
| 없음 | `halfvec(1024)` + HNSW (KURE-v1) |
| `WITH RECURSIVE walk(...)` | **동일** (`AUTOINCREMENT` → `BIGSERIAL`만 변경) |
| `LOG()` | `ln()` |
| `INSERT OR IGNORE` | `INSERT ... ON CONFLICT DO NOTHING` |
| 없음 | RLS 정책 (`chunks_v3`, `graph_nodes`, `graph_edges`) |
| 없음 | Reranker 단계 (RRF 상위 50 → 8) |

⚠️ **RLS 이식 시 주의:** 그래프 확장이 권한 없는 고객사 노드를 경유해 정보를 유추하는 경로가 생기지 않도록, `graph_edges`에도 양단 노드 권한 검증 정책이 필요합니다. PoC에는 권한 개념이 없으므로 이 부분은 별도 설계·침투 테스트 대상입니다.

## 평가셋 형식

```json
[
  {
    "query": "대한항공 홈페이지 관리자 페이지 접속 방법",
    "relevant_docs": ["대한항공_유지보수정보_20260514"],
    "level": "L1",
    "note": "단순 사실검색"
  },
  {
    "query": "코웨이 급여 시스템 장애 대응 절차",
    "relevant_docs": [],
    "level": "L5",
    "note": "정답 없음 — 시스템이 '없음'을 답해야 정답"
  }
]
```

`level`은 기획서 §7.1의 L1~L5 층화를 따릅니다. **L4(그래프 전용)와 L5(정답 없음)를 반드시 포함**해야 그래프 효과와 환각을 각각 측정할 수 있습니다.
