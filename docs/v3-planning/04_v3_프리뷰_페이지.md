# /v3 프리뷰 페이지 — 구현 노트

배포 URL: `https://<서비스>/v3` (로컬은 `http://127.0.0.1:7860/v3`)

기존 v1(`/`)에는 **아무 영향이 없습니다.** 라우트·모듈·프론트엔드 파일이 전부 신규이며, 기존 코드에 대한 변경은 no-cache 미들웨어 조건 한 줄뿐입니다.

---

## 1. 무엇인가

v3 전환의 **UI 시작점**이자, 지금까지 문서로만 있던 진단을 **운영 데이터로 직접 확인**하는 화면입니다.

| 탭 | 상태 | 내용 |
|---|---|---|
| 그래프 탐색 | 동작 | 고객사·문서·청크·엔티티 관계를 force 그래프로. 고객사 클릭 시 2홉 확장, 허브 페널티 on/off 비교 |
| 진단 지표 | 동작 | 노드/엣지 구성, 엔티티 커버리지, 허브 분포, 인덱스 제외 문서 |
| 하이브리드 검색 | 준비 중 | Phase 1 완료 후 (KURE-v1 + FTS + 리랭커) |
| 사전 · 권한 | 준비 중 | Phase 3~4 (엔티티 사전 편집, 고객사별 RBAC) |

---

## 2. 아키텍처 결정 — 스키마 변경 없음

**그래프는 요청 시 운영 코퍼스에서 즉석 생성됩니다.** `graph_nodes` / `graph_edges` 테이블을 만들지 않습니다.

```
GET /api/v3/stats
      │
      ▼
graph_v3.get_graph(_doc_records)
      │
      ├─ 캐시 히트 (지문 동일 + TTL 내) ──▶ 반환      ~9ms
      │
      └─ 미스 ─▶ _doc_records()  (Supabase 또는 파일)
                 ├─ 저장소 메타 문서 제외
                 ├─ 헤딩 보존 청킹 (1200자 / 150자 오버랩)
                 ├─ 규칙 기반 엔티티 추출 (사전 매칭)
                 ├─ 노드·엣지 구성
                 └─ 허브 페널티 계산                   ~60ms
```

**이렇게 한 이유:**

- 마이그레이션 프레임워크가 없는 상태(진단 리포트 §2.8)에서 스키마를 건드리면 롤백이 어렵습니다
- 코퍼스가 135문서라 전체 재빌드가 60ms입니다. 테이블로 얻을 이득이 없습니다
- 항상 최신 코퍼스를 반영합니다. v1/v2 인덱스 드리프트(§2.5)가 구조적으로 불가능합니다
- Phase 4에서 테이블을 도입할 때 `build_graph()` 결과를 그대로 적재하면 됩니다

**제약:** 프로세스별 메모리 캐시이므로 다중 인스턴스에서는 인스턴스마다 따로 빌드합니다. 현 규모에선 무시 가능하고, Phase 4의 테이블 도입이 자연스러운 해법입니다.

## 3. API

| 엔드포인트 | 용도 |
|---|---|
| `GET /v3` | HTML 셸 |
| `GET /api/v3/stats?refresh=0\|1` | 그래프 통계, 고객사 목록, 허브 분포 |
| `GET /api/v3/graph?customers=A,B&maxChunksPerDoc=4` | 시각화용 서브그래프 |
| `GET /api/v3/expand?seeds=&q=&hop=2&hubPenalty=0\|1&intent=` | 그래프 확장 결과 |

응답은 기존 관례를 따릅니다 — 성공은 bare dict, 실패는 `{"error": "..."}` + 상태코드.

### 인증 없는 상태의 방어 장치

검증 과정에서 **인증 없이 코퍼스 본문 표본이 통째로 빠져나가는 경로**가 발견되어 막았습니다. 청크 노드에는 본문 미리보기가 붙어 있고, 청크는 1050자 보폭으로 연속되므로 미리보기를 이어붙이면 각 문서의 상당 부분이 복원됩니다. 초기 구현에서 `?maxChunksPerDoc=0`(무제한으로 해석됨) 한 번으로 **699개 청크 · 본문 85,774자**가 응답됐습니다.

| 방어 | 내용 |
|---|---|
| 고객사 필터 필수 | `customers` 파라미터가 없으면 **청크를 아예 내보내지 않음**(구조만). 검증 결과 본문 0자 |
| `maxChunksPerDoc=0` 의미 수정 | '무제한'이 아니라 '청크 없음' |
| 상한 축소 | `maxChunksPerDoc` 최대 20, `customers` 최대 12곳 |
| 미리보기 축소 | 160자 → **110자** |
| 노드 상한 | `V3_SERIALIZE_MAX_NODES`(기본 1200) 초과 시 절단 + `truncated: true` |
| 재빌드 쿨다운 | `?refresh=1`은 인증 없이 호출 가능하고 Supabase 전체 재조회를 유발하므로 **20초 쿨다운**. 쿨다운 내에는 캐시 반환 |

> 이것들은 **완화책이지 접근 제어가 아닙니다.** `/v3`는 여전히 고객사 목록과 문서 구조를 무인증으로 노출합니다. Phase 3의 RBAC + RLS가 도입되기 전까지는 내부 확인용으로만 쓰는 것을 권고합니다.

### 시드 해소 (`resolve_seeds`)

확장 시드로 고객사 노드가 들어오면 **그 고객사의 청크들로 치환**합니다.

이건 편의 기능이 아니라 정확성 문제입니다. 실제 검색에서 시드는 렉시컬·벡터 상위 **청크**와 링킹된 **엔티티**이지 고객사가 아닙니다. 고객사 노드에서 출발하면 2홉 안에 엔티티를 경유하지 못해 **허브 페널티가 아무 일도 하지 않는 것처럼 보입니다.** 초기 구현에서 실제로 이 버그가 있었고(토글 on/off 결과가 27개로 동일), 시드 해소를 넣고 나서야 효과가 드러났습니다.

## 4. 운영 데이터로 확인된 것

### 허브 페널티 효과 (실측, 2홉 확장)

| 시드 고객사 | 페널티 ON 도달 / 번진 고객사 | 페널티 OFF 도달 / 번진 고객사 |
|---|---|---|
| 대한항공 | **8 / 1곳** | 126 / **38곳** |
| 파크랜드 | **67 / 29곳** | 278 / **53곳** |

페널티가 없으면 대한항공 청크에서 출발한 2홉 확장이 **전체 60개 고객사 중 38곳으로 번집니다.** 사실상 코퍼스를 무차별로 끌어오는 것이며, 검색 결과를 개선하는 게 아니라 희석시킵니다.

> **지표 정정:** 초기에는 '타 고객사 청크 수'를 표시했으나, 확장 결과에서 시드가 제외되므로 **도달 청크는 정의상 전부 타 고객사**가 되어 두 수치가 항상 같았습니다. 정보량이 0인 지표였고 경고 표시가 상시 켜져 있었습니다. **'번진 고객사 수 / 전체'**로 교체했습니다.

### 엔티티 커버리지 정정

PoC에서 보고한 89.8%는 **과대평가였습니다.** `ISSUE_TERMS["기능개선"]`에 `개선·수정·요청·추가·변경` 같은 범용어가 있어 793개 중 639개(81%) 청크에 매칭됐기 때문입니다.

용어를 좁힌 뒤 실제 커버리지는 **42.5%**입니다. 허브 상위도 12.6%(관리자페이지)로 내려가 건강한 분포가 됐습니다.

> **교훈:** 커버리지를 KPI로 삼으면 범용어를 사전에 넣어 숫자를 올리는 유혹이 생깁니다. **허브 엔티티를 제외한 커버리지**를 봐야 하며, 이 화면의 허브 분포 막대가 그 감시 장치입니다.

### 인덱스에서 제외된 문서

저장소 메타 문서 4~5개가 자동 제외됩니다(`EXCLUDED_SOURCES`). 진단 리포트 P0-α 조치가 v3 경로에는 이미 반영되어 있습니다. **v1은 여전히 이 문서들을 인덱싱합니다.**

## 5. 배포

**Render 설정 변경이 필요 없습니다.**

- `Dockerfile:8` `COPY backend/*.py .` → `graph_v3.py` 자동 포함
- `Dockerfile:9` `COPY frontend/ frontend/` → 재귀 복사이므로 `frontend/v3/` 자동 포함
- 신규 의존성 없음 (표준 라이브러리만). `requirements.txt` 변경 없음
- d3는 CDN(`cdnjs.cloudflare.com`)에서 로드

**메모리:** 그래프 로드 후 RSS **약 56MB** (베이스라인 53MB + 그래프 약 14MB). Render 512Mi 대비 여유가 충분합니다.

**검증 완료 항목:** 정적 자산 7종 200 응답, `/api/v3/*` 파라미터 방어(범위 초과 422, 비정수 시드 무시), 동시 요청 20건 오류 0, v1 회귀 없음(`/`, `/api/docs`, `/api/meta`, `/api/search`, `/api/folders` 전부 200), 서버 로그 예외 0건.

### 환경변수 (전부 선택)

| 변수 | 기본값 | 용도 |
|---|---|---|
| `V3_GRAPH_CACHE_TTL` | 300 | 캐시 유효시간(초) |
| `V3_GRAPH_MAX_DOCS` | **1000** | 그래프 빌드 문서 상한 |
| `V3_SERIALIZE_MAX_NODES` | 1200 | 응답 노드 상한 |
| `V3_REBUILD_COOLDOWN` | 20 | 강제 재빌드 최소 간격(초) |
| `V3_CHUNK_MAX_CHARS` | 1200 | 청크 크기 |
| `V3_CHUNK_OVERLAP` | 150 | 청크 오버랩 (강제로 `MAX_CHARS - 100` 이하로 잘림) |
| `V3_PREVIEW_CHARS` | 110 | 청크 미리보기 길이 |

> `V3_GRAPH_MAX_DOCS` 초기값은 3000이었으나 **1000으로 낮췄습니다.** 129문서 그래프가 약 14MB이므로 3000문서면 300MB를 넘어 512Mi 플랜에서 터집니다. 이 상한에 도달하면 메모리 그래프가 아니라 Phase 4의 `graph_nodes`/`graph_edges` 테이블로 가야 한다는 신호입니다.
>
> `V3_CHUNK_OVERLAP >= V3_CHUNK_MAX_CHARS`이면 슬라이딩 윈도우 보폭이 0이 되어 **무한 루프**에 빠지고, 이 루프가 캐시 락을 쥔 채 돌기 때문에 **v1을 포함한 프로세스 전체가 멈춥니다.** 환경변수를 신뢰하지 않고 코드에서 강제로 잘라냅니다.

## 6. 알려진 한계

1. **캔버스는 표본입니다.** 선택한 고객사만, 문서당 청크 4개까지 그립니다. 확장 통계는 전체 그래프 기준이므로 수치와 화면의 강조 표시가 일치하지 않을 수 있습니다. 패널에 이 점을 명시했습니다.
2. **인증이 없습니다.** v1과 동일하게 무인증입니다. `/v3`는 고객사 문서 구조를 한눈에 보여주므로, Render 공개 배포 시 v1보다 노출 범위가 넓다는 점을 인지해야 합니다. Phase 3 이전까지는 내부 확인용으로만 쓰는 것을 권고합니다.
3. **엔티티 사전이 소스코드에 있습니다.** `graph_v3.py`의 `SYSTEM_TERMS` / `ISSUE_TERMS` / `CUSTOMER_ALIASES`. 수정 후에는 `?refresh=1`로 캐시를 비워야 반영됩니다. Phase 4에서 DB + 관리자 UI로 이관합니다.
4. **검색 품질과 무관합니다.** 이 화면은 그래프 **구조**만 보여줍니다. Recall@k·NDCG는 평가셋 구축(P0) 후에야 측정 가능합니다.
5. **CDN 의존.** React·ReactDOM(unpkg), d3(cdnjs)를 런타임에 불러옵니다. d3 로드 실패 시 화면에 명시적 오류를 띄우도록 했지만, 망분리 환경으로 이전할 때는 **세 라이브러리 모두 사내로 반입**해야 합니다.
6. **`hop` 파라미터는 사실상 무력합니다.** 감쇠(0.6)와 허브 페널티가 누적되어 3홉 결과가 2홉과 동일합니다. 프론트엔드는 `hop`을 보내지 않습니다. 3홉 이상이 필요해지면 `min_score` 재조정이 함께 필요합니다.
