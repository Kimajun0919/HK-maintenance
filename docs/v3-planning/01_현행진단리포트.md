# HK-maintenance 현행 시스템 진단 리포트

작성일: 2026-07-20
대상: `backend/` (v1), `v2/` (실험적 BGE-M3 경로)
목적: 온프레미스 Private Graph RAG(v3) 전환에 앞선 현행 병목 진단 및 개선 우선순위 도출

---

## 0. 요약 (Executive Summary)

**결론부터: 현재 검색 품질 저하의 원인은 "그래프가 없어서"가 아닙니다.**

측정된 하이브리드 top-3 정확도는 **0.633**입니다. 이 수치의 원인을 추적한 결과, 배포된 모든 환경에서 임베딩이 **실제 언어모델이 아닌 해시 랜덤 프로젝션**으로 동작하고 있었습니다. 즉 지금까지 "벡터 검색"이라고 불린 것은 의미 기반 검색이 아니었습니다.

이 진단이 중요한 이유는, Graph RAG로 바로 넘어갈 경우 **개선 효과의 귀속(attribution)이 불가능**해지기 때문입니다. 임베딩을 고치기만 해도 상당한 개선이 예상되는데, 이를 그래프 도입의 성과로 오인하면 이후 의사결정이 왜곡됩니다.

**권고 순서:**

| 순위 | 조치 | 예상 효과 | 난이도 |
|---|---|---|---|
| 1 | 실제 임베딩 모델 투입 (KURE-v1) | 높음 | 낮음 |
| 2 | 한국어 형태소 분석 + FTS 정상화 | 높음 | 낮음 |
| 3 | Reranker 도입 | 중~높음 | 낮음 |
| 4 | 평가셋 확충 (30 → 150+) | 측정 가능성 확보 | 중 |
| 5 | 결정론적 엔티티 그래프 | 중 (특정 질의 유형에 집중) | 중 |
| 6 | LLM 기반 관계 추출 그래프 | 불확실 | 높음 |

Graph RAG는 **6번이 아니라 5번**의 형태로 도입해야 합니다. 근거는 3장에서 서술합니다.

## 1. 코퍼스 실측 (규모 인식 교정)

기획의 모든 판단이 여기서 시작되므로 먼저 실측치를 고정합니다.

| 항목 | 실측값 |
|---|---|
| 마크다운 문서 수 | **135개** |
| 전체 텍스트 용량 | **약 400 KB** |
| 고객사 폴더 수 | **60개** (+ 최상위 미분류 문서 = 그래프상 61 Customer 노드) |
| 추정 토큰 수 | **약 10만 토큰** |
| 청크 수 | **793개** (PoC 실측, 1200자/150자 오버랩 기준) |

**이 수치가 의미하는 것:**

- 코퍼스 전체가 **단행본 한 권보다 작습니다.** 일반적인 Graph RAG 문헌이 다루는 규모(수십만~수백만 청크)보다 3~4 자릿수 작습니다.
- 검색 대상이 800개 미만 청크인데 top-3 정확도가 0.633이라는 것은, **후보 집합이 부족한 문제가 아니라 매칭 자체가 틀리고 있다**는 신호입니다.
- 전문 벡터 DB, 분산 그래프 DB, 대형 임베딩 모델(8B급)은 이 규모에서 전부 과잉 설계입니다. **PostgreSQL 단일 스택으로 충분합니다.**
- 반대로 문서 수가 적다는 것은 **문서에 없는 정보는 검색으로 절대 나오지 않는다**는 뜻이기도 합니다. 뒤에서 다룰 "커버리지 문제"의 배경입니다.

## 2. 핵심 병목 진단

### 2.1 [치명] 임베딩이 실제 모델이 아님

**증거:**

| 위치 | 설정 |
|---|---|
| `Dockerfile:20` | `EMBEDDING_BACKEND=hash` |
| `render.yaml:26` | `EMBEDDING_BACKEND=hash` |
| `render.yaml:55` | `EMBEDDING_BACKEND=hash` |
| `start-local.bat:6` | `EMBEDDING_BACKEND=none` |

`hash` 백엔드의 구현은 `backend/rag.py:689-700` — 어휘 토큰에 대한 **blake2b 해시 기반 결정론적 랜덤 프로젝션**입니다. 의미 공간이 아니라 어휘 공간의 축소 사영일 뿐이며, **동의어·유의어·문맥 유사성을 전혀 포착하지 못합니다.**

**결과적으로 현재 시스템은:**

- `vector(384)` 컬럼과 HNSW 코사인 인덱스를 유지하면서
- `settings.json`에서 임베딩에 **0.35의 가중치**를 부여하고
- 그 가중치를 **의미 없는 해시값에 배분**하고 있습니다.

평가 리포트의 `vector_only: 0.467`은 임베딩 모델의 성능이 아니라 **랜덤 프로젝션의 성능**입니다. 이 수치를 근거로 "벡터 검색이 한국어에 약하다"고 판단해서는 안 됩니다.

**부가 문제:** `sentence-transformers`는 `backend/requirements-embeddings.txt`에 분리되어 있고 Docker 이미지에 포함되지 않습니다. 즉 컨테이너에서는 `hash` 외의 선택지가 물리적으로 존재하지 않습니다.

**근본 원인:** Render 무료 플랜의 512Mi 메모리 제약. 이 제약이 `RAG_STARTUP_INDEX=0`, `EMBEDDING_BACKEND=hash`, torch 미포함 등 아키텍처 전반의 우회 설계를 강제했습니다. **온프레미스 전환은 이 제약을 제거하므로, 관련 우회 로직은 이식이 아니라 삭제 대상입니다.**

### 2.2 [치명] Recall 천장 — 정답이 후보에 없음

평가 리포트(`backend/search_quality_report.json`)에서 가장 중요한 패턴:

| 모드 | top-3 | top-5 |
|---|---|---|
| bm25_only | 0.5667 | **0.5667** |
| ngram_only | 0.5667 | **0.5667** |
| vector_only | 0.4667 | **0.4667** |
| hybrid | 0.6333 | **0.6333** |

**모든 모드에서 top-3와 top-5가 완전히 동일합니다.** 4~5위에서 추가로 맞은 케이스가 단 한 건도 없습니다.

이것은 랭킹 문제가 아니라 **회수(recall) 문제**입니다. 순위를 아무리 잘 매겨도 정답 청크가 후보 집합에 애초에 들어오지 않습니다. Reranker를 붙여도 이 30건 중 실패한 11건은 개선되지 않습니다.

**원인 가설 (검증 필요):**

1. 렉시컬 매칭이 `ILIKE '%term%'`이라 형태소 변화형을 못 잡음 (2.3)
2. 청킹 단계에서 정답 구간이 인덱스에서 제외됨 (2.4)
3. 임베딩이 해시라 의미적 우회 경로가 없음 (2.1)
4. **정답이 코퍼스에 실제로 없음** — 이 가능성도 반드시 검증해야 합니다

가설 4가 중요합니다. 평가 케이스에 `"물이 새요"` 같은 매우 일반적인 질의가 포함되어 있는데(top_title이 `기업_리뉴얼`로 완전히 빗나감), 135개 문서 코퍼스에 해당 답변이 존재하지 않는다면 이는 **검색 알고리즘 문제가 아니라 콘텐츠 커버리지 문제**입니다. Graph RAG로는 해결되지 않습니다.

> **선행 과제:** 실패한 11건 각각에 대해 "정답 청크가 DB에 존재하는가"를 수동 확인. 존재하지 않는 건은 평가셋에서 분리하고 콘텐츠 확충 과제로 이관.

### 2.3 [높음] 한국어 처리 부재

**렉시컬 검색이 `ILIKE '%term%'`:**

`backend/storage.py:475-553`, `v2/rag_v2.py:313-373` — 검색어당 7개 컬럼에 대해 부분 문자열 매칭을 수행합니다.

- `tsvector` 없음, GIN 인덱스 없음, 한국어 FTS 설정 없음
- 교착어 변화형 매칭 불가: `"결제"`로 검색 시 `"결제하기"`는 잡히지만 `"결재"`, `"페이먼트"`, `"PG"`는 전혀 안 잡힘
- 비용이 코퍼스 크기에 선형 증가 (현재 규모에선 문제 없으나 확장 시 병목)

**형태소 분석기 미설치:**

`backend/rag.py:147-186`에서 `kiwipiepy`를 optional import 하지만, **어떤 requirements 파일에도 없습니다.** 따라서 운영 환경에서는 항상 정규식 + 문자 n-gram으로 폴백합니다.

한국어에서 형태소 분석 없이 char 2/3-gram에만 의존하면 정밀도가 크게 떨어집니다. 특히 이 코퍼스처럼 **한글·영문·숫자가 혼재된 IT 유지보수 용어**(`SSL 인증서 갱신`, `WAS 재기동`, `DB 커넥션 풀`)에서 문제가 큽니다.

### 2.4 [높음] 청킹이 콘텐츠를 삭제함

`backend/rag.py:313-328` — `split_markdown()` 내부에 **하드코딩된 섹션 제목 스킵 리스트**가 있습니다:

```
문서 개요, 핵심 요약, 상세 내용, ...
```

해당 제목의 섹션은 **인덱스에서 조용히 제외**됩니다. 문제는:

- `핵심 요약`은 이름 그대로 문서의 핵심을 담고 있을 가능성이 높은 섹션입니다
- 제외 사실이 로그에 남지 않아 운영자가 인지할 수 없습니다
- 코퍼스가 400KB밖에 안 되는 상황에서 콘텐츠를 버리는 것은 recall에 직접 타격입니다

유사한 하드코딩 필터가 `is_noise_title_for_answer` (`rag.py:1463`), `is_low_value_context_title` (`rag.py:1524`)에도 존재합니다.

**추가로**, `settings.json:6-8`의 `synonyms` 맵은 **비어 있습니다.** 설정 기반 동의어 처리 경로가 설계되었으나 채워지지 않았고, 대신 도메인 지식이 소스코드에 하드코딩되어 있습니다 (`QUERY_ALIASES`, `INTENT_EXPANSIONS` — `rag.py:47-65`). 8개 고객사 별칭이 파이썬 소스에 박혀 있어 비개발자가 수정할 수 없습니다.

## 3. 그래프 레이어에 대한 냉정한 평가

요구사항이 "Graph RAG 고도화"이므로, 근거를 정확히 정리합니다.

### 3.1 문헌 증거: LLM 구축 그래프는 기본값이 아님

통제된 비교 실험(*RAG vs. GraphRAG: A Systematic Evaluation*, arXiv 2502.11371 — 청킹·임베딩·생성모델을 동일하게 고정)의 결과:

**검색 정확도 (Hotpot / NQ):**

| 방식 | 검색 정확도 |
|---|---|
| **일반 RAG (밀집 검색)** | **0.886 / 0.867** |
| KG-GraphRAG (트리플만) | 0.392 / 0.322 |
| Community-GraphRAG (Local) | 0.675 / 0.422 |

**QA F1 (Llama-3.1-8B/70B):**

| 방식 | NQ (단일 홉) | HotpotQA (멀티 홉) |
|---|---|---|
| **일반 RAG** | **64.78 / 68.18** | 60.04 / 63.88 |
| Community-GraphRAG (Local) | 63.01 / 65.44 | 61.66 / 64.60 |
| Community-GraphRAG (Global) | 54.48 / 55.05 | 45.16 / 46.99 |
| **HippoRAG2** | 61.03 / 64.03 | **63.01 / 64.93** |

**주목할 3가지:**

1. **그래프 불완전성이 근본 제약** — 논문 측정 결과 정답 엔티티의 **약 65.8%만이 구축된 KG에 존재**했습니다. LLM 추출 그래프는 정답의 1/3을 조용히 누락합니다.

2. **Global/Community 검색은 답이 없는 질문에서 환각** — 정답이 코퍼스에 없는 "Null" 질의에서 Community-Global의 정확도는 **19.27% / 13.95%**, 일반 RAG는 **96.01% / 91.36%**. 유지보수 문서처럼 *"매뉴얼에 없습니다"*가 안전상 중요한 답변인 도메인에서는 사실상 실격 사유입니다.

3. **상보성은 실재하나 크기는 제한적** — MultiHop-RAG에서 그래프 전용 정답 13.6%, RAG 전용 정답 11.6%. 두 결과를 **합치는(Integration)** 방식이 최고 단일 방식 대비 **+6.4점**, LLM이 라우팅하는 **선택(Selection)** 방식은 **+1.1점**에 그쳤습니다.

또한 같은 논문이 **LLM-as-a-judge의 심각한 위치 편향**을 문서화했습니다 — 제시 순서만 바꿔도 판정이 완전히 뒤집혔습니다. 시중의 GraphRAG 승률 주장 상당수가 이 방식으로 측정된 것이므로, **v3 평가는 반드시 정답 기반(ground truth) 지표로 설계해야 합니다.**

### 3.2 그럼에도 이 프로젝트에서 그래프가 유효한 이유

위 문헌은 "LLM이 텍스트에서 트리플을 추출해 만든 그래프"에 대한 비판입니다. **본 프로젝트가 필요로 하는 그래프는 성격이 다릅니다.**

HK-maintenance에는 이미 **결정론적이고 신뢰할 수 있는 관계 데이터**가 존재합니다:

| 소스 | 관계 |
|---|---|
| `maintenance_requests` 테이블 | 티켓 ↔ 담당자 ↔ 작업자 ↔ 유형 ↔ 상태 ↔ 일자 ↔ 공수 |
| 폴더 구조 | 문서 ↔ 고객사 (60개 고객사 = 명시적 엔티티) |
| 문서 파일명 규칙 | `_유지보수정보_`, `_HK공통매뉴얼_추가정보_` = 문서 유형 |
| `QUERY_ALIASES` (`rag.py:47-56`) | 고객사 별칭 8건 = 수작업 엔티티 해소 테이블의 원형 |
| 문서 헤딩 계층 | 문서 ↔ 섹션 ↔ 청크 |

**LLM 추출률 65%가 아니라, 100% 정확한 관계가 이미 DB에 있습니다.** 이걸 그래프로 노출하는 것은 위험이 거의 없고 효과가 명확합니다.

**그래프가 실제로 답을 바꾸는 질의 유형:**

- "코웨이 담당자가 처리한 다른 고객사 이슈 중 비슷한 게 있나?"
- "지난 분기 SSL 인증서 관련 접수 건이 가장 많았던 고객사는?"
- "이 고객사에서 과거에 동일 증상이 재발한 이력이 있나?"
- "A 담당자 부재 시 이 고객사 이슈를 아는 사람은?"

이런 질의는 **현재 30건 평가셋에 단 한 건도 없습니다.** 그래프의 가치를 측정할 수단 자체가 없는 상태입니다.

### 3.3 권고: 2단계 그래프 전략

| 단계 | 내용 | 리스크 |
|---|---|---|
| **G1 — 결정론적 그래프** | 관계형 데이터(티켓·담당자·고객사) + 규칙 기반 엔티티(고객사명, 시스템명, 에러코드, 제품명) + 문서 구조를 그래프 테이블로 구축. 벡터 검색으로 시드 청크를 찾고 **1~2홉 확장**으로 이웃을 끌어옴 (Neo4j `VectorCypherRetriever` 패턴) | 낮음. LLM 미사용, 재인덱싱 수 분, 완전 결정론적 |
| **G2 — LLM 보강 그래프** | G1 위에 LLM 추출 트리플을 **추가 엣지로만** 얹음. 텍스트 검색을 절대 대체하지 않음. HippoRAG2 스타일 Personalized PageRank로 확장 | 중. G1 대비 개선폭을 반드시 측정 후 채택 판단 |

**Microsoft GraphRAG의 Global Search는 채택하지 않습니다.** Null 질의 환각률(13.9~19.3%)이 유지보수 문서 도메인에서 수용 불가하고, 커뮤니티 요약 재빌드 비용이 문서 변경 빈도와 맞지 않습니다.

## 4. 개선 우선순위 (근거 기반)

### P0-α — 코퍼스 오염 제거 (반나절, 최우선)

PoC(`v3-poc/`) 실행 중 발견: `organized_maintenance_docs_simple/` 최상위에 **유지보수 콘텐츠가 아닌 저장소 메타 문서 4개**가 인덱싱되고 있습니다.

```
SIMPLIFY_CHANGELOG.md
SIMPLIFY_VALIDATION_REPORT.md
READABILITY_CLEANUP_REPORT.md
READABILITY_FINAL_VALIDATION.md
README.md
HK_CUSTOMER_INFO_INDEX.md      ← 고객사 색인. 노이즈 여부는 판단 필요
```

문서 처리 이력 파일이라 도메인 어휘가 대량 포함되어 **BM25 점수가 비정상적으로 높게 나오며, 여러 질의에서 실제로 렉시컬 상위 1~2위를 차지**했습니다.

135개 중 최소 5개를 제외하는 것만으로 top-3 정확도가 개선될 가능성이 있습니다. **알고리즘 개선보다 훨씬 싼 조치이므로 무엇보다 먼저 확인하십시오.**

### P0 — 측정 가능성 확보 (선행 필수)

| 과제 | 내용 |
|---|---|
| 평가셋 확충 | 30건 → **150건 이상**. 유지보수팀 실제 질의 기반. 4단계 층화: 단순 사실검색 / 복합 추론 / 요약 / **정답 없음(unanswerable)** |
| 실패 케이스 근본 원인 분류 | 현 11건 실패 각각에 대해 "정답이 DB에 존재하는가" 수동 확인. 없으면 콘텐츠 과제로 분리 |
| 그래프 질의 유형 추가 | 3.2의 관계형 질의를 평가셋에 별도 층으로 추가 (없으면 그래프 효과 측정 불가) |
| 지표 정의 | Recall@k, NDCG@k, 정답 기반 F1. **LLM 페어와이즈 판정 금지** (위치 편향) |

> P0 없이 P1 이하를 진행하면 개선 여부를 알 수 없습니다. 이것이 유일한 진짜 선행 조건입니다.

### P1 — 검색 기반 정상화 (효과 대비 비용 최고)

| 과제 | 근거 |
|---|---|
| **실제 임베딩 모델 투입** | `hash` → **KURE-v1** (MIT, 1024d, ~1.2GB FP16). MTEB-ko-retrieval 1위 (Recall@5 0.7385). BGE-M3 파인튜닝 모델 |
| **한국어 형태소 분석 도입** | `kiwipiepy` 정식 의존성화 + 사용자 사전(고객사명, IT 용어). Soynlp로 코퍼스에서 도메인 어휘 1회 마이닝하여 사전 시드 |
| **Postgres 한국어 FTS** | `ILIKE` → Kiwi 표제어 추출 후 `tsvector` + GIN 인덱스. 크래시 세이프, 확장 의존성 없음 |
| **Reranker 도입** | `dragonkue/bge-reranker-v2-m3-ko` (568M, 단일 forward pass, ~1.2GB). 50 후보 리랭크 20~50ms |
| **RRF 융합** | 현재의 고정 가중치 선형 결합 → Reciprocal Rank Fusion. 스케일 불일치 문제 해소 |
| **청킹 스킵 리스트 제거** | `rag.py:313-328`의 하드코딩 제외 목록 삭제. 필요 시 가중치 하향으로 대체 |

> P1만으로 top-3가 어디까지 오르는지가 **v3의 진짜 베이스라인**입니다. 그래프는 이 베이스라인 대비로 평가해야 합니다.

## 5. 이 진단이 v3 설계에 주는 제약

**살릴 것:**

- 청크 ID가 내용 해시 기반(`chunk_id`, `body_hash`)이라 안정적 — 증분 인덱싱에 그대로 활용 가능
- `maintenance_docs` → 하위 테이블 FK CASCADE가 정상 동작
- `maintenance_requests`가 깨끗한 관계형 스파인 — 그래프의 뼈대
- 폴더=고객사 관례가 이미 암묵적 엔티티로 기능 중
- 테이블명 SQL 식별자 검증 로직 (`config.py:69-79`)

**버릴 것:**

- 512Mi 제약이 낳은 모든 우회 설계 (온프레미스에서 무의미)
- `hash` 임베딩 백엔드 전체
- 소스코드 하드코딩 도메인 지식
- 이중 청크 테이블
- 무음 폴백

**새로 만들어야 하는 것:**

- 인증·권한·감사 전 계층 (현재 0%)
- 그래프 스키마 및 인덱싱 파이프라인
- 로컬 LLM 서빙 계층 (현재 LLM은 100% 외부 API + 클라이언트 제공 키)
- 신뢰할 수 있는 평가 하네스 (`backend/eval_retrieval.py:42`에 `TODO: populate cases.json` 잔존)
