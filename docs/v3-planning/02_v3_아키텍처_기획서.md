## 1. 목표와 범위

### 1.1 요구사항 확정 사항

| 항목 | 결정 |
|---|---|
| 마이그레이션 방식 | **v3 신규 설계** (v1 유지·병행 운영, v2는 흡수 후 폐기) |
| 배포 형태 | **온프레미스 / 망분리** |
| 외부 API | **완전 차단** — LLM·임베딩·리랭커 전부 로컬 |
| 권한 | **고객사별 접근 권한 분리** |
| 인증·감사 | **사용자 인증 + 감사 로그 필수** |
| 하드웨어 | 미정 — 본 문서에서 산정 (`03_인프라_구성안.md`) |

### 1.2 설계 원칙

1. **PostgreSQL 단일 스택.** 코퍼스 규모(135문서 / 400KB / ~500청크)에서 전용 벡터 DB·그래프 DB는 과잉 설계입니다. 망분리 환경에서 백업·패치·인증·모니터링 대상을 하나로 유지하는 것이 어떤 벤치마크 이득보다 가치가 큽니다.
2. **결정론 우선.** LLM이 만든 그래프보다 이미 DB에 있는 100% 정확한 관계를 먼저 씁니다.
3. **텍스트를 대체하지 않는 그래프.** 그래프는 항상 원문 청크로 되돌아가는 확장 경로일 뿐, 트리플이 답변 근거가 되지 않습니다.
4. **무음 실패 금지.** 모든 degradation은 명시적 경보 + 응답 메타데이터로 노출합니다.
5. **측정 없이 채택 없음.** 모든 컴포넌트는 이전 베이스라인 대비 정답 기반 지표로 검증 후 채택합니다.

### 1.3 명시적 비목표

- Microsoft GraphRAG의 **Global/Community Search 미채택** — Null 질의 환각률 13.9~19.3%가 유지보수 문서 도메인에서 수용 불가
- **Text2Cypher 미채택** — 분산이 크고 무인증 자연어→쿼리 경로는 망분리 환경 보안 검토가 과함
- 전용 벡터 DB(Milvus/Qdrant), 전용 그래프 DB(Neo4j/FalkorDB) 미도입 — 재검토 조건은 §9

### 2.1 기술 스택 선정

| 계층 | 선정 | 근거 | 대안 |
|---|---|---|---|
| DB | **PostgreSQL 17 + pgvector 0.8** | 기존 자산, RLS로 RBAC 무료 획득, 단일 백업/패치. pgvector 0.8의 iterative index scan이 필터링 시 overfiltering 문제 해소 | Apache AGE 추가 (재귀 CTE가 감당 안 될 때만) |
| 그래프 저장 | **관계형 테이블 (`graph_nodes`/`graph_edges`) + 재귀 CTE** | 예상 노드/엣지 수만 개 수준. 1~3홉 탐색이 ms 단위 | Apache AGE (openCypher, Postgres 확장, 2026-01 RLS 지원 추가) |
| 임베딩 | **nlpai-lab/KURE-v1** (MIT, 1024d, ~568M) | MTEB-ko-retrieval 1위 (Recall@5 0.7385 / NDCG@5 0.6748). BGE-M3 한국어 파인튜닝. 고려대 NLP&AI 연구실 | BAAI/bge-m3 (0.7295, sparse+ColBERT 동시 출력), Qwen3-Embedding-0.6B (한국어 공개 벤치 없음 — 자체 측정 필요) |
| 리랭커 | **dragonkue/bge-reranker-v2-m3-ko** (568M) | 단일 forward pass cross-encoder, 한국어 튜닝, 50후보 20~50ms | BAAI/bge-reranker-v2-m3 (한국어 튜닝판 미달 시) |
| 형태소 | **kiwipiepy** + 사용자 사전 | 활발한 유지보수, pip 설치, C++ 코어, 띄어쓰기 오류·미등록어 강건, 사용자 사전 지원 | MeCab-ko (설치 난이도, 사전 유지보수 저조) |
| 한국어 FTS | **Kiwi 표제어 → `tsvector` + GIN** | 크래시 세이프, 확장 의존성 0, n-gram보다 정밀 | pg_bigm (크래시 세이프, 단순), PGroonga (빠르나 **인덱스가 크래시 세이프 아님** — 망분리 환경에서 REINDEX 런북 부담) |
| LLM 서빙 | **vLLM** | 8 동시 사용자 기준 Ollama 대비 2.2~4배 처리량 (continuous batching + PagedAttention). TGI는 2025-12부터 유지보수 모드 | SGLang (구조화 생성 강점) |
| LLM 모델 | **Qwen3.5-27B (dense), Apache 2.0** | 한국어 벤치 최상위(KMMLU-Pro 73.0, KoBALT 54.9). **라이선스 안전** | HyperCLOVA X SEED Think 32B (NAVER, 상용 무료), Qwen3.5-35B-A3B MoE (처리량 우수) |
| 추출용 소형 LLM | **Qwen3.5-9B** 또는 **SKT A.X 4.0 Light 7B** (Apache 2.0) | 대량 엔티티 추출 배치용. 품질보다 처리량 | — |

### 2.2 ⚠️ 라이선스 경고 — EXAONE

**LG AI Research의 EXAONE 4.5는 비상업 라이선스입니다.** 모델 카드에 *"EXAONE AI Model License Agreement 1.2 - NC"*로 명시되어 있으며, 초기 버전 라이선스는 연구 목적 한정 + 연구 결과 공개 의무까지 포함합니다. **사내 프로덕션 시스템 배포는 LG AI Research와의 별도 상용 계약이 필요합니다.**

한국 온프레미스 LLM 계획에서 가장 자주 놓치는 항목이므로 **PoC 착수 전 법무 검토**를 권고합니다.

부연하면, **LG 자체 기술 리포트 표에서도 Qwen3.5-27B가 EXAONE 4.5 33B를 모든 한국어 벤치마크에서 상회**합니다 (KMMLU-Pro 73.0 vs 67.6, KoBALT 54.9 vs 52.1, KMMMU 51.7 vs 42.7, KRETA 96.5 vs 91.9). 더 작고, Apache 2.0이며, 성능이 높습니다. 한국 모델이 이해관계자 요구사항이라면 **HyperCLOVA X SEED**(상용 무료)가 대안입니다.

### 3.1 온톨로지

### 3.2 노드 타입

| 타입 | 출처 | 신뢰도 | 예상 수 |
|---|---|---|---|
| `Customer` (고객사) | 폴더명 + `maintenance_requests` + 별칭 테이블 | **결정론** | ~60 |
| `Document` (문서) | `maintenance_docs` | **결정론** | ~135 |
| `Chunk` (청크) | 청킹 파이프라인 | **결정론** | ~300–500 |
| `Person` (담당자) | `maintenance_requests.manager_id`, `worker_id` | **결정론** | ~10–30 |
| `Ticket` (접수건) | `maintenance_requests` | **결정론** | CSV 규모에 따름 |
| `System` (시스템) | 규칙 기반 추출 (홈페이지, 그룹웨어, WAS, DB 등) | 규칙 | ~30–80 |
| `IssueType` (이슈유형) | `maintenance_requests.type_id` + 규칙 | **결정론** | ~10–30 |
| `Concept` (개념/용어) | Kiwi 명사 추출 + Soynlp 도메인 어휘 마이닝 | 규칙 | ~500–2000 |

**8개 노드 타입 중 5개가 완전 결정론적입니다.** 이것이 본 설계가 문헌상의 GraphRAG 실패 사례(정답 엔티티의 65%만 추출)를 회피하는 지점입니다.

### 3.3 엣지 타입

| 엣지 | from → to | 출처 | 신뢰도 |
|---|---|---|---|
| `BELONGS_TO` | Document → Customer | 폴더 구조 | 결정론 |
| `HAS_CHUNK` | Document → Chunk | 청킹 | 결정론 |
| `HANDLED` | Person → Ticket | `maintenance_requests` | 결정론 |
| `REQUESTED_BY` | Ticket → Customer | `maintenance_requests` | 결정론 |
| `HAS_ISSUE` | Ticket → IssueType | `maintenance_requests.type_id` | 결정론 |
| `OWNS` | Customer → System | 규칙 추출 + 수동 큐레이션 | 규칙 |
| `MENTIONS` | Chunk → Concept/System/Customer | 사전 기반 매칭 | 규칙 |
| `EVIDENCE_OF` | Chunk → Ticket | 접수건 생성 마크다운 역참조 | 결정론 |
| `SIMILAR_TO` | Chunk ↔ Chunk | 임베딩 코사인 임계값 | 통계 |
| `ALIAS_OF` | Concept → Concept | 별칭 테이블 (기존 `QUERY_ALIASES` 이관) | 큐레이션 |
| `RELATED_TO` | * ↔ * | **G2 단계** LLM 추출 | LLM (보강용) |

### 3.4 물리 스키마

```sql
-- 엔티티 노드
CREATE TABLE graph_nodes (
    node_id      BIGSERIAL PRIMARY KEY,
    node_type    TEXT NOT NULL,          -- Customer/Document/Chunk/Person/...
    natural_key  TEXT NOT NULL,          -- 타입 내 고유 키 (예: 고객사명, chunk_id)
    label        TEXT NOT NULL,          -- 표시명
    properties   JSONB NOT NULL DEFAULT '{}',
    customer_id  BIGINT REFERENCES customers(customer_id),  -- RLS용 비정규화 (NULL = 공통자료)
    confidence   REAL NOT NULL DEFAULT 1.0,
    source       TEXT NOT NULL,          -- 'relational' | 'rule' | 'llm' | 'manual'
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (node_type, natural_key)
);
CREATE INDEX ON graph_nodes (node_type);
CREATE INDEX ON graph_nodes (customer_id);
CREATE INDEX ON graph_nodes USING GIN (properties);

-- 엔티티 별칭 (기존 QUERY_ALIASES 하드코딩 대체)
CREATE TABLE graph_node_aliases (
    alias        TEXT NOT NULL,
    normalized   TEXT NOT NULL,          -- Kiwi 정규화 형태
    node_id      BIGINT NOT NULL REFERENCES graph_nodes(node_id) ON DELETE CASCADE,
    weight       REAL NOT NULL DEFAULT 1.0,
    PRIMARY KEY (alias, node_id)
);
CREATE INDEX ON graph_node_aliases (normalized);

-- 관계 엣지
-- 고객사 정준 테이블 (RLS 및 FK 의 기준점)
-- graph_nodes 의 Customer 노드와 1:1 대응하되, 권한 판정은 항상 이 테이블 기준
CREATE TABLE customers (
    customer_id  BIGSERIAL PRIMARY KEY,
    code         TEXT UNIQUE NOT NULL,       -- 폴더명 등 안정 식별자
    name         TEXT NOT NULL,
    is_active    BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE TABLE graph_edges (
    edge_id      BIGSERIAL PRIMARY KEY,
    src_id       BIGINT NOT NULL REFERENCES graph_nodes(node_id) ON DELETE CASCADE,
    dst_id       BIGINT NOT NULL REFERENCES graph_nodes(node_id) ON DELETE CASCADE,
    edge_type    TEXT NOT NULL,
    weight       REAL NOT NULL DEFAULT 1.0,
    -- 허브 페널티 반영 확장 가중치. 인덱싱 시 미리 계산 (§4.2 참조)
    exp_weight   REAL NOT NULL DEFAULT 1.0,
    -- dst 노드의 출차수. exp_weight 재계산용 캐시
    dst_degree   INT NOT NULL DEFAULT 1,
    confidence   REAL NOT NULL DEFAULT 1.0,
    source       TEXT NOT NULL,
    properties   JSONB NOT NULL DEFAULT '{}',
    UNIQUE (src_id, dst_id, edge_type)
);
CREATE INDEX ON graph_edges (src_id, edge_type);
CREATE INDEX ON graph_edges (dst_id, edge_type);   -- 역방향 탐색

-- v3 청크 (단일 테이블, 단일 writer)
CREATE TABLE chunks_v3 (
    chunk_id      TEXT PRIMARY KEY,
    document_id   TEXT NOT NULL,
    customer_id   BIGINT REFERENCES customers(customer_id),  -- RLS 키. NULL = 공통자료
    ordinal       INT NOT NULL,
    heading_path  TEXT[],                        -- 계층 경로
    body          TEXT NOT NULL,
    body_hash     TEXT NOT NULL,                 -- 증분 인덱싱 키
    lemmas        TEXT NOT NULL,                 -- Kiwi 표제어 (FTS 소스)
    fts           TSVECTOR GENERATED ALWAYS AS (to_tsvector('simple', lemmas)) STORED,
    embedding     halfvec(1024),                 -- KURE-v1, 저장 절반
    token_count   INT,
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON chunks_v3 USING GIN (fts);
CREATE INDEX ON chunks_v3 USING hnsw (embedding halfvec_cosine_ops);
CREATE INDEX ON chunks_v3 (customer_id);
CREATE INDEX ON chunks_v3 (document_id, ordinal);
```

**설계 노트:**

- `halfvec(1024)` — pgvector 0.8의 half-precision. 인덱스 크기 절반, 이 규모에서 품질 손실 무시 가능
- `fts`는 **GENERATED 컬럼** — `lemmas` 갱신 시 자동 동기화. v1/v2 드리프트 문제의 구조적 방지
- `customer_id`를 청크·노드에 비정규화 — RLS 정책이 조인 없이 동작하도록
- `body_hash` 유지 — 기존 증분 인덱싱 자산 승계
- `graph_edges`에 양방향 인덱스 — 재귀 CTE 성능

## 4. 검색 파이프라인

### 4.1 전체 흐름

```
질의
 │
 ├─▶ [1] 질의 정규화
 │      ├ Kiwi 형태소 분석 → 표제어
 │      ├ 별칭 해소 (graph_node_aliases)
 │      └ 엔티티 링킹 → 시드 노드 집합 E₀
 │
 ├─▶ [2] 병렬 1차 검색 (각 top-30)
 │      ├ A. 렉시컬:  tsvector @@ query  (GIN)
 │      ├ B. 벡터:    KURE-v1 임베딩 KNN (HNSW)
 │      └ C. 그래프:  E₀ 시드 → 1~2홉 확장 → 도달 청크
 │
 ├─▶ [3] RRF 융합  score = Σ 1/(k + rank_i),  k=60
 │      → 상위 50 후보
 │
 ├─▶ [4] Reranker (bge-reranker-v2-m3-ko)
 │      → 상위 8
 │
 ├─▶ [5] 컨텍스트 구성
 │      ├ 청크 원문 + 헤딩 경로
 │      ├ 그래프 컨텍스트 (관련 고객사/티켓/담당자 사실)
 │      └ 토큰 예산 제어
 │
 └─▶ [6] 답변 합성 (vLLM / Qwen3.5-27B)
        ├ 근거 필수 프롬프트
        ├ 인용 청크 ID 반환
        └ 근거 부족 시 명시적 "확인 불가" 응답
```

### 4.2 그래프 확장 검색 (핵심 신규 로직)

Neo4j의 `VectorCypherRetriever` 패턴을 Postgres 재귀 CTE로 구현합니다. **"벡터로 시드를 찾고, 그래프로 이웃을 끌어온다"** — 복잡도 대비 효과가 GraphRAG 영역에서 가장 높은 아이디어입니다.

```sql
-- 시드 노드에서 최대 2홉 확장하여 도달 가능한 청크 회수
WITH RECURSIVE walk AS (
    SELECT node_id, 0 AS hop, 1.0::real AS decay
    FROM graph_nodes
    WHERE node_id = ANY($1)               -- 시드 노드 (엔티티 링킹 + 벡터 top-k의 청크)
  UNION ALL
    -- ⚠️ 0.6::real 캐스팅 필수. 무타입 0.6 은 double precision 으로 해석되어
    --    재귀 UNION 의 타입 불일치 에러("recursive query column has type real
    --    in non-recursive term but type double precision overall")를 유발함
    -- ⚠️ e.weight 가 아니라 e.exp_weight (허브 페널티 반영분)를 사용
    SELECT e.dst_id, w.hop + 1, w.decay * e.exp_weight * 0.6::real
    FROM walk w
    JOIN graph_edges e ON e.src_id = w.node_id
    WHERE w.hop < 2
      AND e.edge_type = ANY($2)                     -- 허용 엣지 타입 (질의 의도별)
      AND w.decay * e.exp_weight * 0.6::real > 0.05 -- 가지치기
),
reached AS (
    SELECT node_id, MAX(decay) AS score
    FROM walk GROUP BY node_id
)
SELECT c.chunk_id, c.body, r.score
FROM reached r
JOIN graph_nodes n ON n.node_id = r.node_id AND n.node_type = 'Chunk'
JOIN chunks_v3 c ON c.chunk_id = n.natural_key
ORDER BY r.score DESC
LIMIT 30;
```

#### PoC 검증 결과 반영 (필독)

`v3-poc/`에서 실제 코퍼스로 검증한 결과, **엣지 화이트리스트만으로는 부족했습니다.** 초기 구현에서 그래프 확장이 상위 8개에 새 청크를 **0개** 추가했습니다.

원인: `SSL인증서` 같은 엔티티가 수백 청크에 연결된 **허브 노드**가 되어, 이를 경유한 2홉 확장이 코퍼스 상당 부분을 균일 점수로 끌어옵니다. 그래프 런의 순위가 무의미해지고 RRF에서 렉시컬을 희석시키기만 합니다.

**추가로 반드시 필요한 2가지:**

**(1) 허브 페널티** — 엔티티 차수에 로그 역비례하는 감쇠 (IDF와 동일 발상). 인덱싱 시 미리 계산해 엣지에 저장:

```sql
-- 인덱싱 종료 시 1회 실행. dst_degree 갱신 후 exp_weight 재계산
UPDATE graph_edges e SET dst_degree = COALESCE(d.deg, 1)
FROM (SELECT src_id, COUNT(*)::int AS deg FROM graph_edges GROUP BY src_id) d
WHERE d.src_id = e.dst_id;

UPDATE graph_edges
SET exp_weight = weight / (1 + 0.9 * ln(GREATEST(dst_degree, 1)::real))::real;
```

**(2) 고객사 스코핑을 별도 런으로 분리** — 유지보수 도메인에서 "어느 고객사인가"는 거의 항상 1차 필터입니다. 이 신호를 일반 그래프 확장 점수에 묻히게 두면 안 됩니다. RRF에서 가장 높은 가중치(1.2) 부여.

실측 효과 — 질의 `대한항공 홈페이지 관리자 페이지`:

| 순위 | 렉시컬 단독 | 그래프 + 스코핑 |
|---|---|---|
| 1 | `_공통 / SIMPLIFY_CHANGELOG` ❌ | `대한항공 / 디비_접속_정보` ★ |
| 2 | `_공통 / SIMPLIFY_CHANGELOG` ❌ | `대한항공 / HK공통매뉴얼` |
| 3~6 | 대한항공 문서 | 대한항공 문서 |

★ = 렉시컬이 전혀 찾지 못한 청크

> **설계 원칙:** 그래프 확장은 "켜면 좋아지는 기능"이 아닙니다. **엣지 화이트리스트 + 허브 페널티 + 도메인 스코핑**이 함께 있어야 작동하며, 셋 중 하나라도 없으면 순수한 손해입니다.

**의도별 엣지 화이트리스트** — 무분별한 확장은 노이즈를 늘리므로 질의 의도에 따라 허용 엣지를 제한합니다:

| 의도 | 허용 엣지 |
|---|---|
| 고객사 정보 조회 | `BELONGS_TO`, `HAS_CHUNK`, `OWNS` |
| 이력/유사사례 | `REQUESTED_BY`, `HAS_ISSUE`, `EVIDENCE_OF`, `SIMILAR_TO` |
| 담당자 조회 | `HANDLED`, `REQUESTED_BY`, `ASSIGNED_TO` |
| 일반 매뉴얼 | `MENTIONS`, `HAS_CHUNK`, `ALIAS_OF` |

### 4.3 융합을 RRF로 바꾸는 이유

현재는 `settings.json`의 고정 선형 가중치(`bm25 0.35 / ngram 0.15 / embedding 0.35 / ...`)로 **점수를 직접 합산**합니다. BM25 점수, 코사인 유사도, ILIKE 매칭 수는 스케일이 전혀 다르므로 이 합산은 의미가 불분명합니다.

RRF는 **점수가 아닌 순위**를 융합하므로 스케일 문제가 원천적으로 없고, 이질적 검색기 3종(렉시컬/벡터/그래프)을 섞기에 적합합니다.

```
RRF(d) = Σ_i  w_i / (60 + rank_i(d))
```

`w_i`는 검색기별 가중치로, 평가셋 기반 그리드 서치로 결정합니다.

### 4.4 답변 없음(Unanswerable) 처리

문헌상 GraphRAG Global Search의 최대 약점이 정답 없는 질문에 대한 환각(정확도 13.9~19.3%)이었습니다. 유지보수 문서 도메인에서 *"매뉴얼에 없습니다"*는 안전 관련 정답입니다.

**방어 3중화:**

1. Reranker 최고점이 임계값(예: 0.3) 미만이면 LLM 호출 없이 "관련 문서 없음" 반환
2. 프롬프트에 근거 인용 강제 — 인용 불가 시 명시적 거부
3. 응답 후처리 — 인용된 청크 ID가 실제 컨텍스트에 없으면 경고 플래그

평가셋에 **의도적 unanswerable 케이스를 20% 이상** 포함하여 이 경로를 상시 측정합니다.

### 5.2 고객사별 접근 권한 (RBAC + RLS)

```sql
CREATE TABLE users (
    user_id      BIGSERIAL PRIMARY KEY,
    username     TEXT UNIQUE NOT NULL,
    display_name TEXT NOT NULL,
    password_hash TEXT NOT NULL,
    role         TEXT NOT NULL,   -- 'admin' | 'staff' | 'viewer'
    is_active    BOOLEAN NOT NULL DEFAULT TRUE,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 사용자 ↔ 고객사 접근 권한
CREATE TABLE user_customer_access (
    user_id     BIGINT REFERENCES users(user_id) ON DELETE CASCADE,
    customer_id BIGINT NOT NULL,
    permission  TEXT NOT NULL DEFAULT 'read',  -- read | write
    PRIMARY KEY (user_id, customer_id)
);

-- ★ 역할 분리 필수: 스키마 소유자와 애플리케이션 접속 역할을 분리한다
--   RLS 는 테이블 소유자에게 적용되지 않으므로, 앱이 소유자로 접속하면
--   모든 정책이 조용히 무시된다. (본 설계에서 가장 놓치기 쉬운 함정)
CREATE ROLE hk_owner NOLOGIN;                 -- 스키마/DDL 소유
CREATE ROLE hk_app   LOGIN PASSWORD :'app_pw';-- 앱 접속 (테이블 미소유)
CREATE ROLE hk_index LOGIN PASSWORD :'idx_pw';-- 인덱싱 워커 (쓰기)
GRANT SELECT                        ON ALL TABLES IN SCHEMA public TO hk_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO hk_index;

-- 행 수준 보안: 요청 세션에 app.current_user_id 설정 후 자동 필터링
ALTER TABLE chunks_v3 ENABLE ROW LEVEL SECURITY;
ALTER TABLE chunks_v3 FORCE  ROW LEVEL SECURITY;   -- ★ 소유자에게도 강제

-- 읽기 정책
CREATE POLICY chunk_read ON chunks_v3
FOR SELECT USING (
    customer_id IS NULL                                    -- 공통자료는 전원 열람
    OR current_setting('app.role', true) = 'admin'
    OR customer_id IN (
        SELECT uca.customer_id FROM user_customer_access uca
        WHERE uca.user_id = NULLIF(current_setting('app.current_user_id', true), '')::bigint
    )
);

-- 쓰기 정책 (없으면 인덱싱 워커의 INSERT/UPDATE 가 전부 거부됨)
CREATE POLICY chunk_write ON chunks_v3
FOR ALL TO hk_index USING (true) WITH CHECK (true);

-- graph_nodes / graph_edges 에도 동일 패턴 적용.
-- graph_edges 는 양단 노드 권한을 모두 검사해야 함 (§5.2 주의 참조)
```

> ⚠️ **`FORCE ROW LEVEL SECURITY`와 역할 분리가 없으면 RLS는 아무것도 막지 못합니다.** `ENABLE`만으로는 테이블 소유자에게 정책이 적용되지 않으므로, 앱이 `POSTGRES_USER`(= 스키마 소유자)로 접속하는 흔한 구성에서는 모든 정책이 **조용히 무시**됩니다. 진단 리포트가 지적한 "무음 실패"가 보안 계층에서 재현되는 최악의 사례이므로, Phase 3에서 **RLS 우회 여부를 실제 계정으로 검증**하는 테스트를 반드시 포함하십시오.

**RLS를 쓰는 이유:** 애플리케이션 코드에 `WHERE customer_id IN (...)`를 흩뿌리면 **한 군데만 빠뜨려도 정보 유출**입니다. RLS는 DB가 강제하므로 새 쿼리를 추가해도 자동으로 보호됩니다. 그래프 확장 검색처럼 재귀 CTE가 여러 테이블을 도는 경우 특히 중요합니다.

> ⚠️ **주의:** 그래프 확장이 권한 없는 고객사 노드를 경유해 정보를 유추하게 하면 안 됩니다. `graph_edges`도 양 끝 노드 권한을 확인하도록 정책을 걸고, 확장 결과에 대해 **애플리케이션 레벨 2차 검증**을 병행합니다.

## 6. 인덱싱 파이프라인

```
문서 입력 (MD/DOCX/XLSX/PDF) 또는 CSV 접수내역
   │
   ├─▶ 파싱 → 마크다운 정규화
   │     (PDF는 텍스트 추출 실패 시 OCR 폴백 — 2단계)
   │
   ├─▶ 구조 인식 청킹
   │     ├ 헤딩 계층 보존 (heading_path)
   │     ├ 800~1200자 목표, 150자 오버랩
   │     ├ 표는 분할하지 않음
   │     └ ❌ 하드코딩 섹션 스킵 리스트 제거
   │
   ├─▶ Kiwi 형태소 분석 → lemmas 컬럼 (FTS 소스)
   │
   ├─▶ 엔티티 추출 (규칙 기반)
   │     ├ 고객사: 폴더 + 별칭 테이블 사전 매칭
   │     ├ 시스템/제품: 큐레이션 사전
   │     ├ 에러코드/버전: 정규식
   │     └ 개념어: Kiwi 명사 + Soynlp 마이닝 사전
   │
   ├─▶ 그래프 upsert (graph_nodes / graph_edges)
   │
   ├─▶ 임베딩 (KURE-v1, 배치)
   │     └ body_hash 변경분만 재계산
   │
   └─▶ 트랜잭션 커밋 (청크 + 그래프 원자적 갱신)
```

**단일 writer 원칙.** v1/v2 드리프트의 재발을 막기 위해 청크·FTS·벡터·그래프를 **하나의 트랜잭션**에서 갱신합니다. `fts` 컬럼이 GENERATED이므로 구조적으로 어긋날 수 없습니다.

**증분 인덱싱.** `body_hash` 비교로 변경 청크만 재임베딩. 135문서 전체 재빌드도 GPU에서 수 초, CPU에서도 수 분 수준이므로 전체 재빌드 옵션도 상시 유지합니다.

## 7. 평가 하네스

### 7.1 평가셋 설계 (P0 — 최우선)

현행 30건 → **150건 이상**. GraphRAG-Bench의 4단계 층화를 차용하되 도메인에 맞게 조정:

| 층 | 비중 | 예시 |
|---|---|---|
| L1 단순 사실검색 | 30% | "코웨이 홈페이지 도메인 만료일은?" |
| L2 복합 추론 (멀티홉) | 25% | "SSL 갱신이 필요한 고객사 중 담당자가 A인 곳은?" |
| L3 요약/종합 | 15% | "이 고객사 유지보수 이력 요약해줘" |
| **L4 그래프 전용** | 10% | "이 담당자가 처리한 유사 이슈가 다른 고객사에 있나?" |
| **L5 정답 없음** | **20%** | "코웨이 급여 시스템 장애 대응 절차는?" (해당 문서 부재) |

L4와 L5가 신규입니다. **L4 없이는 그래프의 가치를 측정할 수 없고, L5 없이는 환각을 측정할 수 없습니다.**

### 7.2 지표

| 지표 | 대상 |
|---|---|
| Recall@5, Recall@10 | 검색 (recall 천장 문제 직접 추적) |
| NDCG@5 | 랭킹 |
| Answer F1 (정답 대비) | 생성 |
| **Abstention Accuracy** | L5 층 — 올바르게 "없음"을 답한 비율 |
| Citation Precision | 인용 청크가 실제 근거인 비율 |
| p50 / p95 지연 | 성능 |

**LLM 페어와이즈 판정은 사용하지 않습니다.** 통제 실험에서 제시 순서만 바꿔도 판정이 뒤집히는 심각한 위치 편향이 확인되었습니다. 정답 기반 지표만 사용합니다.

### 7.3 단계별 게이트

각 단계는 이전 단계 대비 **측정된 개선**이 있을 때만 다음으로 진행합니다.

| 게이트 | 조건 |
|---|---|
| G-0 → G-1 | 평가셋 150건 구축 완료, 기존 실패 11건 근본원인 분류 완료 |
| G-1 → G-2 | P1(임베딩+형태소+FTS+리랭커) 적용 후 Recall@5 유의미 상승, L5 Abstention ≥ 0.8 |
| G-2 → G-3 | G1 그래프 적용 후 **L4 층에서** 명확한 개선. L1~L3 회귀 없음 |
| G-3 → G-4 | G2(LLM 보강) 적용 후 G1 대비 개선. 개선 없으면 **채택하지 않음** |

> G-3 → G-4에서 "채택하지 않음"이 정당한 결론입니다. 문헌 증거상 LLM 추출 그래프가 개선을 못 낼 가능성이 실재합니다.

### 8.1 v1 병행 운영 전략

- v1은 Phase 4 완료까지 현행 유지 (Render + Supabase)
- v3는 별도 온프레미스 인스턴스로 구축, 동일 코퍼스 동기화
- Phase 4 게이트 통과 후 사용자 그룹 단위 점진 전환
- v2 코드는 Phase 1에서 v3에 흡수 후 폐기 (`v2/rag_v2.py`의 BGE-M3 경로 지식은 v3에 승계)

## 9. 리스크 및 재검토 조건

| 리스크 | 영향 | 대응 |
|---|---|---|
| **실패 케이스가 콘텐츠 부재 때문** | Graph RAG로 해결 불가. 프로젝트 전제 붕괴 | Phase 0에서 최우선 검증. 콘텐츠 확충을 별도 과제로 분리 |
| **L4 그래프 질의 수요가 실제로 적음** | 그래프 투자 대비 효과 미미 | Phase 0에서 유지보수팀 실제 질의 로그·인터뷰로 수요 검증 |
| EXAONE 등 모델 라이선스 위반 | 법적 리스크 | Phase 0 법무 검토. Apache 2.0 모델(Qwen3.5) 기본 채택 |
| GPU 조달 지연·가격 | 일정 지연 | 2026년 GPU 시장 왜곡 심각(§인프라 문서). 조기 발주, CPU 폴백 경로 확보 |
| RLS 성능 저하 | 응답 지연 | 코퍼스 규모상 영향 미미 예상. Phase 3에서 측정 |
| **그래프 확장이 권한 우회 경로 생성** | 정보 유출 | 엣지 양단 권한 검증 + 애플리케이션 2차 검증 + 침투 테스트 |
| 개발 리소스 부족 | 일정 전반 | Phase 1까지만 해도 상당한 개선. 단계별 독립 가치 확보 설계 |

### 재검토 조건 (설계 전환 트리거)

| 조건 | 전환 |
|---|---|
| 문서 수 > 10,000 또는 청크 > 100k | 전용 벡터 DB 검토 |
| 그래프 엣지 > 500만, 3홉 이상 탐색 상시 필요 | Apache AGE 또는 Neo4j 검토 |
| 재귀 CTE p95 > 200ms | Apache AGE 도입 |
| 동시 사용자 > 100 | vLLM 다중 인스턴스 / GPU 증설 |
| 다국어(영문 등) 요구 발생 | BGE-M3로 임베딩 교체 검토 |

> **Neo4j Community는 채택 대상이 아닙니다.** RBAC·클러스터링이 없고(본 프로젝트 핵심 요구사항), 공식 지원 정책상 **보안 패치 보장이 없습니다**. Enterprise는 코어당 연 수천 달러 수준으로 알려져 있어(벤더 미공개, 제3자 집계) 135문서 규모에 정당화되지 않습니다.

## 10. 즉시 착수 가능한 항목 (Quick Wins)

하드웨어 도착 전에도 진행 가능하며, v1에도 즉시 적용 가능한 것들:

1. **`rag.py:313-328` 청킹 스킵 리스트 제거** — 코드 한 곳, 즉시 recall 개선 가능성
2. **평가 케이스 11건 실패 원인 수동 분류** — 방향 결정에 필수, 반나절 작업
3. **`kiwipiepy`를 requirements에 정식 추가** — 이미 코드에 있으나 설치가 안 됨
4. **Soynlp로 코퍼스 도메인 어휘 마이닝** — 1일 작업, 사전 시드 확보
5. **`QUERY_ALIASES` 하드코딩 → 설정 파일 이관** — `settings.json`의 빈 `synonyms` 맵 활용
6. **무음 예외 처리에 로깅 추가** — `storage.py:298,376`, `v2/rag_v2.py:104,492,377,392`
7. **모델 라이선스 법무 검토 착수** — 리드타임이 길 수 있음
