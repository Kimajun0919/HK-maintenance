작성일: 2026-07-20
전제: 망분리 (외부 인터넷 차단), 사내 서버실 설치

## 1. 규모 산정 근거

| 항목 | 실측/추정 |
|---|---|
| 문서 수 | 135개 (마크다운) |
| 코퍼스 용량 | 약 400 KB |
| 청크 수 | **793** (PoC 실측) |
| 고객사 수 | 60 (폴더 기준) |
| 동시 사용자 | **확인 필요** — 본 산정은 10~30명 가정 |

### 2.3 ⚠️ 조달 시 반드시 확인할 2가지

**(1) GeForce 데이터센터 EULA 제약**

NVIDIA 드라이버 라이선스는 **GeForce 카드의 데이터센터 배포를 금지**합니다. RTX 5090을 사내 서버실 랙에 설치하는 것은 라이선스 위반 소지가 있습니다. A6000/L40S/RTX PRO 6000은 데이터센터 라이선스가 포함됩니다. **감사 대응이 필요한 조직이라면 약 30% 프리미엄을 지불하고 프로 카드를 선택하는 것이 정당합니다.**

**(2) 2026년 GPU 시장 가격 왜곡**

현재 GPU 시장은 심각하게 왜곡되어 있습니다:

- RTX 5090: 출시가 $1,999 → 현재 $4,180~4,200 (**MSRP의 2배 이상**)
- RTX PRO 6000 Blackwell: 2025년 예약가 ~$7,673 → 현재 $13,250 (**73% 상승**)
- 국내 다나와 기준 RTX 5090 ₩649만~665만 (현금가 ₩550만~600만)

**예산에 충분한 버퍼를 두고, 조기 발주를 권고합니다.** 정가 기준 계획은 실패합니다.

## 3. 소프트웨어 스택

### 3.1 Docker Compose 구성

```yaml
# docker-compose.yml (구성 개요 — 실제 배포 시 시크릿/볼륨 경로 조정 필요)
services:

  postgres:
    image: pgvector/pgvector:pg17          # 내부 레지스트리로 미러링 필요
    environment:
      POSTGRES_DB: hk_maintenance
      POSTGRES_USER: hk_app
      POSTGRES_PASSWORD_FILE: /run/secrets/pg_password
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./initdb:/docker-entrypoint-initdb.d:ro
    command: >
      postgres
      -c shared_buffers=8GB
      -c work_mem=64MB
      -c maintenance_work_mem=2GB
      -c max_parallel_workers_per_gather=4
      -c log_statement=ddl
    networks: [internal]
    restart: unless-stopped

  vllm:
    image: vllm/vllm-openai:latest         # 내부 레지스트리 미러
    command: >
      --model /models/Qwen3.5-27B-AWQ
      --quantization awq_marlin
      --max-model-len 32768
      --gpu-memory-utilization 0.75
      --served-model-name hk-llm
      --disable-log-requests
    environment:
      HF_HUB_OFFLINE: "1"                  # 망분리: 런타임 다운로드 차단
      TRANSFORMERS_OFFLINE: "1"
    volumes:
      - /srv/models:/models:ro
    deploy:
      resources:
        reservations:
          devices: [{driver: nvidia, count: 1, capabilities: [gpu]}]
    networks: [internal]
    restart: unless-stopped

  embedder:
    build: ./services/embedder             # KURE-v1 + bge-reranker-v2-m3-ko
    environment:
      EMBED_MODEL: /models/KURE-v1
      RERANK_MODEL: /models/bge-reranker-v2-m3-ko
      HF_HUB_OFFLINE: "1"
      CUDA_VISIBLE_DEVICES: "0"            # vLLM과 동일 GPU 공유
    volumes:
      - /srv/models:/models:ro
    deploy:
      resources:
        reservations:
          devices: [{driver: nvidia, count: 1, capabilities: [gpu]}]
    networks: [internal]
    restart: unless-stopped

  api:
    build: ./backend-v3
    environment:
      DATABASE_URL: postgresql://hk_app@postgres/hk_maintenance
      LLM_BASE_URL: http://vllm:8000/v1
      EMBED_URL: http://embedder:8080
      APP_ENV: production
    depends_on: [postgres, vllm, embedder]
    networks: [internal, edge]
    restart: unless-stopped

  indexer:
    build: ./backend-v3
    command: python -m indexer.worker
    environment:
      DATABASE_URL: postgresql://hk_app@postgres/hk_maintenance
      EMBED_URL: http://embedder:8080
    volumes:
      - /srv/documents:/documents:ro       # 문서 반입 디렉터리
    depends_on: [postgres, embedder]
    networks: [internal]
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    ports: ["443:443"]
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./certs:/etc/nginx/certs:ro        # 사내 CA 발급 인증서
    depends_on: [api]
    networks: [edge]
    restart: unless-stopped

networks:
  internal:
    internal: true          # ★ 외부 통신 불가 — 망분리 핵심
  edge:

volumes:
  pgdata:
```

**핵심 설계점:**

- `internal: true` — DB·LLM·임베더가 속한 네트워크는 **외부로 나갈 수 없습니다.** Docker 레벨의 망분리 강제
- `HF_HUB_OFFLINE=1`, `TRANSFORMERS_OFFLINE=1` — 런타임 모델 다운로드 시도 자체를 차단
- vLLM과 embedder가 **동일 GPU 공유** — `gpu-memory-utilization 0.75`로 vLLM에 상한을 두고 나머지를 임베더/리랭커(합계 ~2.5GB)에 할당
- `nginx`만 `edge` 네트워크에 노출 — TLS 종단 단일 지점

### 3.2 GPU 메모리 배분 (L40S 48GB 기준)

| 대상 | 할당 |
|---|---|
| Qwen3.5-27B AWQ INT4 가중치 | ~17 GB |
| vLLM KV 캐시 (32k 컨텍스트) | ~19 GB |
| KURE-v1 (FP16) | ~1.2 GB |
| bge-reranker-v2-m3-ko (FP16) | ~1.2 GB |
| CUDA 컨텍스트·단편화 여유 | ~6 GB |
| **합계** | **~44.4 / 48 GB** |

FP8로 올릴 경우 가중치가 ~33GB로 늘어 KV 캐시가 압박받으므로, **INT4(AWQ) + 넉넉한 KV 캐시**가 동시성 관점에서 유리합니다. 품질 차이는 평가셋으로 검증 후 결정하십시오.

## 4. CPU 전용 가능성 검토

**결론: 검색 스택은 CPU 가능, 생성은 불가.**

| 컴포넌트 | CPU 실현성 |
|---|---|
| KURE-v1 임베딩 (568M) | ✅ 가능. 500청크 배치 인덱싱 수 분. 질의 임베딩 수십 ms |
| bge-reranker (568M) | ✅ 가능. 50후보 리랭크 수백 ms (GPU 대비 10배 느리나 수용 가능) |
| Kiwi 형태소 분석 | ✅ 문제없음 (C++ 코어) |
| PostgreSQL + pgvector | ✅ 문제없음 (500청크는 인덱스도 불필요한 규모) |
| **LLM 생성 (27B)** | ❌ **불가** |

**LLM 생성이 CPU에서 불가한 이유는 구조적입니다.** 병목은 연산이 아니라 **메모리 대역폭**입니다:

- DDR5-6400 다채널: 약 50~100 GB/s
- GDDR7 (RTX 5090): 1,800 GB/s
- **10~30배 격차이며 소프트웨어로 해소 불가능**

실측 보고: 최신 Xeon(AMX 지원) 또는 EPYC Genoa에서 7B~13B 모델이 **단일 스트림 8~25 tok/s**. 56코어 Xeon에서 Mistral-24B가 짧은 프롬프트에 ~4초 지연. **동시 사용자 10~30명에서는 CPU 배치 효율이 급격히 떨어져 응답 시간이 분 단위가 됩니다.**

### 4.1 절충안 — 하이브리드

GPU 조달이 지연될 경우의 임시 운영안:

| 계층 | 배치 |
|---|---|
| 인덱싱·임베딩·리랭킹 | CPU (야간 배치, 지연 무관) |
| 검색 (렉시컬+벡터+그래프) | CPU (충분) |
| **답변 생성** | 사내 별도 LLM 서버 호출, 또는 **생성 없이 검색 결과만 제공** |

GPU가 실제로 필요한 것은 **Phase 2의 vLLM 서빙 하나뿐**입니다. Phase 1(검색 정상화), Phase 3(보안), Phase 4(그래프)는 전부 GPU 없이 진행 가능하며, Phase 1의 임베딩·리랭킹도 CPU로 개발·검증할 수 있습니다(운영 성능은 GPU 필요).

따라서 **GPU 조달 리드타임과 Phase 1·3·4 개발을 병행**시키고, Phase 2만 하드웨어 입고에 맞춰 배치하는 것이 합리적입니다.

## 5. 망분리 운영 체계

망분리 환경의 실질적 난점은 성능이 아니라 **"인터넷 없이 어떻게 유지보수하는가"**입니다. 반드시 사전 설계해야 합니다.

| 항목 | 조치 |
|---|---|
| **컨테이너 이미지** | 내부 레지스트리(Harbor 등) 구축 또는 `docker save`/`load` 오프라인 번들. `pgvector/pgvector:pg17`, `vllm/vllm-openai` 등 사전 반입 |
| **Python 패키지** | 내부 PyPI 미러(devpi/Nexus) 또는 `pip download` 오프라인 wheel 번들 + 해시 고정 |
| **모델 가중치** | Qwen3.5-27B-AWQ(~17GB), KURE-v1(~1.2GB), bge-reranker-ko(~1.2GB), Kiwi 모델 → `/srv/models`에 사전 배치. 체크섬 검증 |
| **Kiwi 사용자 사전** | 버전 관리 대상. 관리자 UI로 갱신 가능하게 설계 |
| **보안 패치** | 정기 오프라인 패치 반입 프로세스 정의 필수. PostgreSQL·OS·컨테이너 베이스 이미지 CVE 추적 담당자 지정 |
| **TLS 인증서** | 사내 CA 발급. 갱신 주기 캘린더 등록 |
| **시각 동기화** | 사내 NTP 서버 (감사 로그 신뢰성) |
| **반입 매체 통제** | 문서·모델 반입 경로 및 승인 절차 사내 정책에 따름 |

> **가장 흔한 실패 지점:** 초기 구축은 성공했으나 6개월 뒤 보안 패치를 적용할 방법이 없는 상태. **패치 반입 절차를 Phase 2 산출물에 포함**시키십시오.

## 6. 백업 및 재해복구

| 대상 | 방식 | 주기 | 보존 |
|---|---|---|---|
| PostgreSQL | `pg_dump` 논리 백업 + WAL 아카이빙 | 일 1회 전체 / WAL 연속 | 30일 |
| 원본 문서 | `/srv/documents` rsync | 일 1회 | 90일 |
| 모델 가중치 | 1회 백업 (불변) | 변경 시 | 영구 |
| 설정·시크릿 | 별도 안전 보관 (Git + 암호화) | 변경 시 | 영구 |
| 감사 로그 | 별도 아카이브 (append-only) | 월 파티션 | 사내 정책 |

**RTO/RPO 목표(안):** RTO 4시간, RPO 24시간. 사내 정책에 맞춰 확정 필요.

**복구 검증:** 분기 1회 백업으로부터 실제 복원 리허설. 검증 없는 백업은 백업이 아닙니다.

**단일 서버 구성의 한계를 명시합니다.** 본 안은 단일 노드이므로 하드웨어 장애 시 서비스 중단됩니다. 가용성 요구가 높다면 (a) 예비 서버 콜드 스탠바이, (b) Postgres 스트리밍 복제 + 2노드 구성을 추가 검토해야 하며, GPU 이중화는 비용상 권하지 않습니다.

## 7. 모니터링

| 대상 | 지표 |
|---|---|
| GPU | 사용률, VRAM, 온도, 전력 (`nvidia-smi` exporter) |
| vLLM | 요청 큐 길이, 토큰 처리량, TTFT, 배치 크기 |
| API | 요청률, p50/p95/p99 지연, 에러율, 검색 단계별 소요 |
| PostgreSQL | 커넥션, 느린 쿼리, 인덱스 히트율, 디스크 |
| **검색 품질** | 리랭커 최고점 분포, "결과 없음" 비율, 인용 실패율 |
| **보안** | 로그인 실패, 권한 거부, 비정상 질의 패턴 |

**Prometheus + Grafana**를 내부에 구성하되, 망분리이므로 알림은 사내 메일/메신저 게이트웨이 경유로 설계합니다.

**특히 중요:** 현행 시스템의 최대 문제 중 하나가 무음 실패였습니다(진단 리포트 §2.6). v3에서는 다음을 **반드시 경보 대상**으로 지정합니다:

- 임베딩 서비스 응답 실패 → 렉시컬 폴백 발생
- pgvector 인덱스 미사용 (seq scan 감지)
- 리랭커 미동작
- 그래프 확장 0건 반환 지속
